Problem Statement — Unverified Postgres INSERT Ceiling for audit_receipts

The current Vyomanaut audit architecture assumes that a single PostgreSQL instance can sustain the write throughput generated by the audit_receipts table under production-scale audit load. However, this assumption has not yet been validated against the real production schema and workload characteristics.

This creates a critical scalability and reliability risk because the audit system depends entirely on timely receipt insertion to validate provider behavior and maintain accurate reliability scoring.

Why This Is a Problem

The audit_receipts table is not a lightweight benchmark table.
Each audit operation generates database writes that include:

large binary signature fields,
unique nonce validation,
row-level security checks,
complex consistency constraints,
and a two-phase INSERT → UPDATE workflow.

These characteristics significantly increase write overhead compared to generic PostgreSQL benchmark workloads.

As provider count and chunk count grow, audit traffic scales linearly:

Audit writes/day = Providers × Chunks per provider

Example:

1,000 providers × 10,000 chunks
= 10,000,000 audit rows/day
≈ 116 INSERTs/sec continuously

Because the architecture performs both:

a Phase-1 INSERT (PENDING state), and
a Phase-2 UPDATE (finalized receipt),

the effective write load is approximately doubled.

Root Cause

The currently referenced PostgreSQL throughput estimate (~5,000–10,000 INSERTs/sec) is derived from generic benchmark literature and not from the actual Vyomanaut schema.

The production schema introduces several performance-heavy components:

1. Wide Row Structure

Each row stores:

provider signatures,
service signatures,
response hashes,
timestamps,
audit metadata.

This increases row size and reduces page density.

2. Unique Index Validation

The table enforces uniqueness on challenge_nonce.

Every INSERT therefore requires:

a B-tree lookup,
uniqueness verification,
and index maintenance.

As the table grows into billions of rows, index lookups become progressively slower.

3. Row-Level Security (RLS)

RLS policies are enabled on the table.

Although lightweight individually, policy evaluation adds overhead to every row operation.

4. Complex CHECK Constraints

Consistency constraints are evaluated during writes to ensure valid audit state transitions.

These checks increase CPU cost per INSERT/UPDATE.

5. Two-Phase Write Workflow

Each audit receipt requires:

an initial INSERT,
followed by an UPDATE after countersigning.

This effectively doubles write amplification.

Potential Impact

If the actual sustainable INSERT ceiling is lower than expected, several cascading failures may occur:

Increased Write Latency

Audit responses begin queueing while waiting for database writes.

False TIMEOUT Results

Providers may respond correctly within network deadlines, but delayed database writes can cause audits to be recorded as TIMEOUT failures.

Incorrect Reliability Scoring

False TIMEOUTs reduce provider reliability scores despite correct behavior.

Provider Penalties and Churn

Healthy providers may:

lose earnings,
receive lower release multipliers,
or leave the network entirely.
Repair and Availability Pressure

Provider churn increases repair workload and may reduce overall network reliability.

Long-Term Scalability Concern

The audit_receipts table is append-only and continuously grows over time.

Estimated growth:

~10 million rows/day
~3.6 billion rows/year
~1–2 TB/year depending on row width and indexing

As table and index size increase:

cache efficiency decreases,
index lookups become disk-bound,
and write throughput degrades further.

This means the INSERT ceiling is not static — it may decline over time as the system ages.

Current Gap in Research

The architecture documents acknowledge that benchmarking is required, but:

no production-equivalent benchmark has been defined,
no latency acceptance criteria exist,
and no verified sustainable throughput number has been established.

As a result, one of the system’s most critical operational assumptions remains unverified.
