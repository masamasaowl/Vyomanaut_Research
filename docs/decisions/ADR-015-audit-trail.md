# ADR-015 — Signed Receipt Exchange + Transparent Merkle Log

**Status:** Accepted
**Topic:** #2 Proof of Storage (audit trail component)
**Supersedes:** —
**Superseded by:** —
**Research source:** Papers 07, 11

---

## Context

Blockchain was the trust anchor in all surveyed DSNs. It provided: (1) an immutable audit log, (2) automatic payment triggers on verified proof, and (3) public dispute resolution. We replace blockchain with a non-crypto mechanism that provides the same guarantees without on-chain writes.

## Decision

**V2: Signed Receipt Exchange**

Every audit event produces two signed artefacts:

1. **Provider receipt:**
   ```
   {chunk_id, audit_challenge, response_hash, timestamp}
   signed with the provider's registered private key
   ```
   Proves: "I responded to this challenge at this time."
   Stored: locally in a receipt store on the provider's machine.

2. **Microservice countersignature:**
   ```
   {provider_receipt_hash, pass/fail, timestamp}
   signed with the microservice private key
   ```
   Proves: "We recorded this result."
   Stored: in an append-only Postgres table (INSERT only, no UPDATE/DELETE, enforced at DB level).

**Flow:** Provider sends response → microservice validates → microservice inserts one row → microservice returns countersignature. Provider's receipt is only considered valid after receiving the countersignature.

**Known limitation:** The microservice may crash between validation and countersignature. Fix: microservice writes to an append-only Kafka log before sending the countersignature. Provider's receipt is held pending until the log entry is confirmed.

**Receipt replay attack prevention:** Response hash is `SHA256(chunk_id || challenge_nonce || response_hash || timestamp)` — not just response_hash alone.

**Clock skew attack prevention:** Challenge nonce is `HMAC(server_secret, chunk_id + server_timestamp)` where server_timestamp is generated by the microservice, not the provider.

**V3: Transparent Merkle Log (upgrade path)**

Build the Merkle tree from a timestamped snapshot (not a live query — like Kafka compaction). Any provider can run an inclusion proof against the published daily root — the same guarantee Certificate Transparency provides for TLS certificates. This replaces blockchain's public dispute resolution function.

Note: Merkle Log may not immediately convince NAS providers who expect blockchain-equivalent guarantees. This is a sales and trust challenge, not a technical one.

## Consequences

**Positive:**
- Coordination required is very low — only one DB write per audit
- Signed receipts give providers a verifiable record they can use in disputes
- Append-only Postgres table is resistant to tampering (no UPDATE/DELETE at DB level)

**Negative / trade-offs:**
- The microservice is a single point of failure for countersignature issuance
- No public verifiability in V2 (requires V3 Merkle Log upgrade)
- Providers must trust the microservice in V2
