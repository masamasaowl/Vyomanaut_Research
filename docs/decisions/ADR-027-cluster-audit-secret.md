# ADR-027 — Cluster-Shared Audit Secret: HKDF Derivation and Rotation

**Status:** Accepted
**Topic:** #1 Coordination Architecture, #2 Proof of Storage
**Supersedes:** —
**Superseded by:** —
**Research source:** ADR-017 (challenge nonce design), ADR-025 (microservice cluster),
Paper 17 — RFC 8439 (HKDF / key derivation), Paper 11 — Bailis (distributed state management)

---

## Context

ADR-017 specifies that every audit challenge nonce is:

```
challenge_nonce = HMAC(server_secret, chunk_id + server_ts)
```

ADR-025 specifies a (3,2,2) quorum cluster of N=3 microservice replicas that each issue
challenge requests independently.

The contradiction: if Replica A generates a nonce using its local `server_secret` and then
crashes before recording the receipt, and Replica B handles the retry, Replica B cannot validate
the provider's response because it was computed against Replica A's secret. This is not an edge
case — replica failure is an expected operational event in a (3,2,2) cluster.

If a provider is audited during a replica failover, the retry challenge is issued by a different
replica with a different (or no) shared secret. The provider's response is unverifiable. The
receipt is logged as FAIL. Over time, every provider audited during a failover accumulates
systematic false FAIL events. At daily audits across 10,000 providers, failovers produce
hundreds of wrongful FAILs per month.

No ADR specifies how server_secret is distributed, stored, or rotated across replicas.

---

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| Per-replica unique secret | No distribution problem | Challenge-response is invalid across replica boundaries; every failover produces false FAILs |
| Shared secret via secrets manager (loaded at startup) | Simple, correct | Rotation requires all replicas to reload simultaneously; brief window of mixed-version nonces |
| **Derived from cluster master seed via HKDF with version tag** | Rotation is graceful; old version honoured for 24h overlap; correct across all replicas at all times | Requires a secrets manager for the cluster master seed; slightly more code |

---

## Decision

### 1. Derivation

The operational `server_secret` is derived per-version from a single `cluster_master_seed`:

```
server_secret_v{N} = HKDF-SHA256(
  ikm  = cluster_master_seed,          // 32 bytes, generated once at cluster bootstrap
  salt = cluster_id,                   // UUID, stable per deployment — domain separation
  info = "vyomanaut-audit-secret-v" + N,
  len  = 32
)
```

`cluster_master_seed` is:
- 32 bytes generated by the operator at cluster bootstrap via `vyomanaut-admin bootstrap-secret`
- Stored in a secrets manager (HashiCorp Vault, AWS SSM Parameter Store, or equivalent)
- Loaded into memory by each replica at startup; **never written to disk in plaintext**
- All three replicas read from the identical secrets manager path

This approach is analogous to Tahoe-LAFS's capability model ([Paper 18](../research/paper-18-tahoe-lafs.md)): a single root secret
deterministically derives all operational keys. One master seed → all versioned server secrets.

### 2. Challenge nonce format

Every challenge nonce is prefixed with a 1-byte version identifier so any replica can select
the correct server_secret for validation:

```
challenge_nonce = version_byte (1B) || HMAC-SHA256(server_secret_v{N}, chunk_id + server_ts)
                = 33 bytes total
```

The version byte increments only on rotation (see §4). In steady state it is constant.

The `audit_receipts` schema (ADR-017) does not need a new column: the version byte is embedded
in the `challenge_nonce BYTEA(32)` field — update this column to `BYTEA(33)`.

### 3. Validation across replicas

When Replica B receives a provider's response to a challenge originally issued by Replica A:

1. Replica B reads `version_byte` from the `challenge_nonce` prefix.
2. Replica B loads `server_secret_v{version_byte}` from the secrets manager (or in-memory cache
   refreshed every 5 minutes).
3. Replica B recomputes the expected nonce and validates the provider's `response_hash`.
4. The receipt is written as normal.

No cross-replica coordination is needed during validation. Every replica can independently
validate any nonce as long as they share the same versioned secret — which they do, because
they all derive from the same `cluster_master_seed`.

### 4. Rotation protocol

Rotation is needed if the cluster master seed is suspected of compromise, or on a scheduled
basis (suggested annually).

```
Step 1: Operator generates new cluster_master_seed or increments version N → N+1
         (incrementing version with same seed is sufficient for scheduled rotation)
Step 2: Store server_secret_v{N+1} in the secrets manager under the new version path
Step 3: All replicas pick up v{N+1} at their next 5-minute secrets cache refresh
Step 4: For 24 hours after rotation:
         - All replicas accept nonces under version N (old) and version N+1 (new)
         - New challenges are issued under version N+1
Step 5: After 24 hours, retire version N
         - Any in-flight challenge under version N has had a full audit cycle (24h) to complete
         - No valid in-flight challenge can reference a version older than 24 hours
Step 6: Remove server_secret_v{N} from the secrets manager
```

The 24-hour overlap window is consistent with the audit cycle length. A challenge issued in the
last second before rotation will be responded to within one polling cycle. No provider is
wrongfully penalised during rotation.

### 5. Secrets manager operational requirement

The secrets manager must:
- Support per-path versioning (path `/vyomanaut/audit-secret/v{N}`)
- Be reachable by all three microservice replicas
- Support access control (only microservice replicas can read the secret paths)
- Be part of the disaster recovery runbook — if the secrets manager is unavailable, replicas
  cannot start and audit challenges halt (acceptable: better than issuing unverifiable challenges)

Fallback for local development and simulation mode ([ADR-029](./ADR-029-bootstrap-minimum-viable-network.md)): the cluster master seed may be
passed as an environment variable (`VYOMANAUT_CLUSTER_MASTER_SEED`). This is explicitly
prohibited in production deployments.

---

## Consequences

**Positive:**
- Replica failover no longer produces false FAIL audit receipts
- Rotation is graceful with a 24-hour overlap; no provider is penalised during rotation
- The HKDF derivation pattern is consistent with how all other Vyomanaut keys are derived
  (ADR-019, ADR-020) — one master → many operational keys, downward derivation only

**Negative / trade-offs:**
- A secrets manager is a new operational dependency; it must be included in the deployment
  runbook and availability SLA
- If the secrets manager is unreachable at replica startup, the replica cannot derive
  `server_secret` and will not serve audit challenges — this is the correct failure mode
  (fail closed) but adds operational complexity
- The `challenge_nonce` column in `audit_receipts` expands from 32 to 33 bytes — negligible
  storage impact but requires a schema migration before V2 launch

**Open constraints:**
- The secrets manager product choice (Vault vs AWS SSM vs GCP Secret Manager) is an
  infrastructure decision deferred to the deployment team; any of the three is acceptable
  provided it supports versioned paths and IAM-gated access
- Secrets manager HA requirements (it must not be a single point of failure for audit
  operations) must be specified in the deployment runbook
